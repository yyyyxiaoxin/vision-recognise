# 视觉与云台对接同步说明

## 1. 当前结论

主控侧基础框架已经稳定，当前与视觉/云台相关的部分已经具备以下条件：

- `NAV / ADV` 主控状态机已完成
- 视觉 UART 协议已跑通
- UART1 接收并发问题已修复
- `NAV / ADV` 原有流程实测通过
- 灰度与底盘准备层已完成，不会阻塞你这边继续推进云台方向

也就是说，你现在可以把主要精力放在：

- 视觉识别稳定性
- 云台真实驱动
- ADV 模式扫描闭环联调

## 2. 当前主控已经固定的视觉协议

主控当前识别文本帧格式为：

```text
$SHAPE,TARGET,VALID\r\n
```

已支持的映射关系：

- `CIRCLE -> A`
- `TRIANGLE -> B`
- `RECT -> C`
- `PENTAGON -> D`
- `NONE -> 无目标`

示例：

```text
$CIRCLE,A,1\r\n
$TRIANGLE,B,1\r\n
$RECT,C,1\r\n
$PENTAGON,D,1\r\n
$NONE,X,0\r\n
```

当前主控不会改变这套协议，请你后续视觉输出继续保持一致。

## 3. 当前主控相关引脚

视觉串口：

- `VISION_UART = UART1`
- `PB6 = TX`
- `PB7 = RX`
- `9600 8N1`

当前 UART1 RX 已采用明确单一路径处理，不再存在“中断收字节”和“主循环轮询收字节”并发喂同一解析器的问题。

## 4. 当前模式语义

### NAV

流程是：

- 用户切到 `NAV`
- 长按启动
- 主控进入固定检测等待
- 等待视觉帧
- 收到有效目标后进入蜂鸣与运行流程

你这边对 `NAV` 的要求：

- 只要按协议发送正确帧即可
- 不需要处理云台扫描

### ADV

当前流程是：

- 用户切到 `ADV`
- 长按启动
- 主控进入 `GSCAN`
- 当前 `gimbal.c/h` 仍为占位实现
- 收到有效视觉帧后，主控停止扫描占位流程并继续后续运行

这意味着：

- 主控的 ADV 流程外壳已经在
- 但真实“云台扫描动作”还没有完成
- 这部分后续需要你这边的无刷云台方案落地后，再与主控联调

## 5. 当前主控对云台的期待边界

主控侧已经预留的接口思路是：

- `gimbal_init()`
- `gimbal_center()`
- `gimbal_scan_start()`
- `gimbal_scan_step()`
- `gimbal_stop()`
- `gimbal_is_scanning()`

目前它们还是占位层，不代表真实无刷云台驱动已经完成。

## 6. 你这边下一步最需要明确的事

请尽快确认以下问题：

1. 云台执行器具体型号是什么
   - 无刷云台电机
   - 配套驱动板
   - 控制方式

2. 云台是几维
   - 一维水平扫描
   - 还是水平 + 俯仰二维

3. 控制信号准备怎么给 MSPM0
   - PWM
   - 串口
   - 其他驱动接口

4. 近期阶段目标是否先只做：
   - ADV 模式扫描动作
   - 不做闭环瞄准

如果这些还没定，就不能直接进入真实云台实现。

## 7. 当前建议分工边界

你这边负责：

- 视觉识别稳定性
- 指示图形识别正确率
- 无刷云台驱动
- 扫描动作设计
- 后续 ADV 扫描与识别联调

主控侧负责：

- 模式切换
- 状态机调度
- 启动/蜂鸣/流程控制
- 视觉结果接收
- 与底盘/云台的系统级串联

## 8. 当前你不需要担心的部分

这些已经稳定，不是你当前阻塞项：

- 视觉 UART 协议
- 主控状态机主干
- NAV 流程
- ADV 基础等待流程
- 灰度与底盘桥接准备层

## 9. 你下一步最推荐的开发顺序

建议按下面顺序推进：

1. 固定视觉输出协议不变
2. 明确无刷云台硬件方案
3. 先做最小扫描动作
4. 再做 ADV 模式下“扫描中识别到目标即停止”
5. 最后再讨论是否需要闭环瞄准
